How IRSA Works

EKS OIDC Infrastructure
▸EKS OIDC Provider
•URL: oidc.eks.{region}.amazonaws.com/id/{cluster-id}
•Publishes /.well-known/openid-configuration and JWKS endpoint
•Registered with AWS IAM as a trusted identity provider
▸Kube API Server
•Configured with --service-account-issuer (OIDC URL)
•Issues JWTs via Projected ServiceAccount Token Volume
•Tokens contain: iss, sub, aud claims
▸Pod Identity Webhook (EKS Managed)
•Mounts Projected ServiceAccount Token Volume into pod
•Sets AWS_ROLE_ARN and AWS_WEB_IDENTITY_TOKEN_FILE env vars
•AWS SDKs automatically call STS AssumeRoleWithWebIdentity

The Problem for Domino

Dynamic ServiceAccounts vs. Static IAM Trust Policies
▸All IRSA infrastructure is already in place on EKS
•OIDC provider, webhook, and token projection are built-in
▸The core challenge: IRSA binds roles to ServiceAccount names
•Domino dynamically generates SA names per execution (e.g. run-12345-abcde)
•IAM trust policies require exact SA name matching
•Would need constant trust policy updates — impossible to maintain
▸Our Goal
•Leverage existing EKS IRSA machinery with minimum Domino interventions
•Just one mutation: swap the SA name to a stable, username-based SA
4 / 10

Keycloak-Based User-to-Role Mapping

Keycloak Groups as the Source of Truth
▸Challenge: Map Domino users to AWS IAM roles with dynamic SAs
•ServiceAccounts are dynamically created per execution
•User identity must persist across sessions
▸Solution: Keycloak groups encode IAM role assignments
•Groups in path /irsa-mappings/<role-arn> hold members
•User JWT contains group memberships — no external deps needed
▸Benefits
•Manageable without AWS console access
•IaC compatible — version-controlled via kc_apply_group_spec.py
•Dry-run support: preview changes before applying
5 / 10

Version-Controlled, Auditable Configuration
Group Specification (groups.json)

  "groups": [
    { "name": "arn:aws:iam::123456789:role/data-scientist-role", "members": ["alice", "bob"] },
    { "name": "arn:aws:iam::123456789:role/ml-engineer-role",    "members": ["charlie", "alice"] }
  ]
}
CLI Usage (sync.py)
# Apply Keycloak group spec (dry-run first)
python kc_apply_group_spec.py --spec groups.json --dry-run
python kc_apply_group_spec.py --spec groups.json

# Sync users to K8s ServiceAccounts
python sync.py users --default-aws-role-arn $ROLE --dry-run
python sync.py users --default-aws-role-arn $ROLE

# Sync Keycloak groups to IAM trust policies
python sync.py irsa --dry-run
python sync.py irsa

The Mutation Webhook

Dynamic → Fixed Username-Based ServiceAccounts

❌ Before (Dynamic)

  • SA Name: run-12345-abcde
  • Trust Policy: Must update per run
  • IAM Binding: Impossible to maintain
  • Audit: "Which user was run-12345?"
→

✅ After (Username-Based)

  • SA Name: alice
  • Trust Policy: Fixed — sub: ...alice
  • IAM Binding: Stable, user-centric
  • Audit: Clear — "alice"
💡 The sub claim never changes for user alice regardless of how many executions they run. Trust policy uses: "sub": "system:serviceaccount:*:alice"

Architecture Summary

End-to-End Integration
▸Three core integrations: Keycloak ↔ K8s API (SAs) ↔ AWS IAM (Roles)
▸Admin Service (FastAPI) — Central orchestration layer
•POST /k8s/users/sync — Create/update user ServiceAccounts
•POST /iam/irsa/sync — Sync trust policies
•GET /keycloak/groups, POST /keycloak/groups/member
▸Three client interfaces
•CLI (sync.py) — with --dry-run support
•Streamlit UI (app.py) — Visual admin with User Mappings, IRSA Groups, Sync tabs
•Mutation Webhook — Swaps SA name at pod creation time
▸Admin service uses IRSA itself — fully bootstrapped, no static credentials
